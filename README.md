## CS-320

### How can I ensure that my code, program, or software is functional and secure?
To ensure the development of functional and secure code, I strive to test the solutions I develop against suites of tests that validate correct operation, stress the solution under repetitive operations, as well as attempt to break the functionality in as many ways as the compiler will allow.
In implying each of these testing strategies, I can first prove that the solutions meets or exceeds the determined user requirements of the solution. Secondly, I prove that the system can perform reliably under stress and with repetitive use as would be seen in real world scenarios. Lastly, I find myself needing to be creative in inventing ways to break what I've built so as to prove system resiliance and redundancy to faults and attacks, or even accidental misuse.

### How do I interpret user needs and incorporate them into a program?
Interpreting user needs is a dynamic task that varies per project. Though these needs are often understood as requirements, they can translate into functional, non-functional, security oriented requirements, or even UI/UX graphical design outlines. Incorporating them into a program requires more than just interpretation of system requirements, but an understanding of what the user desires in their experience. Where a requirement may be to display a short description of a search query of database, and may describe what information should be displayed, it may not describe what the graphical theme of the display should be or the font of the text of the results that may be most fitting with the product's theme or that may be most easily readable by the user.

### How do I approach designing software?
My approach to designing software is to first understand what the solution should do. From there, I can build an outline in pseudocode or in diagrams of how the system should accomplish it's goals. Once a functional outline is complete, I work on the MVP (Minimum Viable Product), building with the goal to expand upon later. Following OOP principles, I build the MVP to be the back-end root of the system, something the user never directly sees but will be what the solution runs on. Once complete, I test and debug from there. Once the core of the solution is proven to work, I build the outter layer of the application that involves either the UI or the API calls, depending on the system design this can either be new code built into existing classes, or additional classes that call on the core functional classes, connecting the user to the system, and so on and so forth. I continue this iteratively until the project is retired, as development never concludes since software always tends to have room for improvement.
